## 基本原则

围绕协作者必须回答的问题组织文档：

1. 这是什么项目？
2. 怎样运行和验证？
3. 应该在哪里修改？
4. 哪些行为不能意外改变？
5. 依赖哪些数据和配置？
6. 怎样部署和恢复？
7. 修改代码后必须同步哪份文档？

优先建立小而清晰的导航体系，不追求内容很多但互相重复。稳定知识放进主题文档，有时间属性的历史放进 changelog 或 ADR。

## 文档选择

| 仓库信号 | 创建或维护 | 不需要创建的情况 |
|---|---|---|
| 有人需要发现或运行项目 | `README.md` | 活跃仓库不能省略 |
| 多名协作者或 coding agent 修改仓库 | `AGENTS.md` 或 `CONTRIBUTING.md` | 上级目录已有作用范围明确的同类规则 |
| 存在不明显的模块边界或技术约束 | 架构文档 | 只是用途单一的简单脚本，README 足够说明 |
| 存在业务不变量、权限、价格、导出或流程 | 业务规则文档 | 只有基础 CRUD，没有领域规则 |
| 存在持久化实体或迁移 | 数据模型文档 | 不保存持久数据 |
| 存在运行时或构建环境变量 | 配置文档和环境变量示例 | 只有一两个明显的非敏感变量，README 足够说明 |
| 应用需要部署或日常运维 | 运维/runbook 文档 | 仅在本地运行，没有运维流程 |
| 非技术用户需要执行多步骤流程 | 用户指南 | 只有开发者使用 |
| 项目发生任何实质写入 | changelog | 纯只读调查或没有产生仓库改动的操作 |
| 重要技术选择需要保留取舍理由 | ADR | 决策常规、容易撤销或从代码即可看出 |
| 需要并行开发、自动部署或多人协作 | Git 协作说明 | 项目只是一次性实验且不会继续维护 |

不要强制使用这些文件名。现有文档约定清楚一致时，应继续沿用。

所有持续维护的项目都必须至少具备人类入口、协作入口和 changelog。简单项目可以把内容压缩到少量文件，例如 `README.md`、`AGENTS.md` 和 `CHANGELOG.md`；复杂项目可以使用 `docs/` 主题文档和 changelog 索引。最低要求是强制的，文件数量和组织方式应保持最小。

## 单一事实来源

为每类事实指定一个长期负责人：

| 事实 | 长期负责人 | 支撑证据 |
|---|---|---|
| 安装命令和首次运行 | README | 依赖清单、lockfile、开发配置 |
| 模块边界和请求流程 | 架构文档 | import、路由注册、依赖关系 |
| 权限和业务不变量 | 业务规则 | middleware、领域函数、测试 |
| 实体字段和关系 | 数据模型 | schema、迁移、序列化、store |
| 环境变量和默认值 | 配置文档 | 环境变量示例、配置加载器、工作流 |
| 部署、备份、回滚和健康检查 | 运维文档 | CI/CD、脚本、进程管理配置 |
| 最终用户操作 | 用户指南 | 实际界面和授权行为 |
| branch、PR、worktree 和发布流程 | Git 协作说明 | GitHub 设置、CI/CD、仓库约定 |
| 某一次历史修改 | changelog | Git diff、issue、release |
| 决策理由 | ADR | 候选方案和决策背景 |

其他文档需要提到同一事实时，只保留帮助读者找到负责文档的摘要和链接。

文档与仓库不一致时，按以下证据顺序判断：

1. 可执行代码和 schema。
2. 测试和自动化。
3. 当前配置示例。
4. 长期主题文档。
5. changelog、旧笔记和聊天记录。

不要把仓库行为等同于生产环境状态。生产环境可能存在不同配置或未进入仓库的运维变化，必须明确这个边界。

## 新协作者设计

假设参与者并不知道：

- 使用哪个 package manager 和 runtime 版本。
- 前端与后端是否需要分别启动。
- 环境变量文件从哪里获得。
- 本地使用哪种数据库模式才安全。
- 权限、branch、review、部署和回滚怎样工作。
- branch 和 worktree 有什么区别。
- 哪些操作会破坏数据或影响生产环境。

提供下一步具体操作、预期结果和失败后的恢复路径。简要解释陌生命令；除非项目读者确实需要，否则不要把标准工具写成冗长教程。

使用“修改前阅读”路径，不要要求所有人一次读完全部文档：

| 修改类型 | 阅读并同步 |
|---|---|
| UI 行为 | 架构、业务规则、用户指南 |
| API 或模块边界 | 架构文档 |
| 权限或校验 | 业务规则 |
| 数据库字段 | 数据模型、迁移说明 |
| 环境变量 | 环境变量示例、配置文档 |
| 部署流程 | 运维文档 |
| branch 或发布规则 | Git 协作说明、CI/CD |

## 迁移策略

多个旧文件互相竞争时：

1. 盘点独有事实和重复事实。
2. 根据代码或自动化验证每条重要事实。
3. 把每条事实移动到唯一的长期负责人。
4. 删除旧路径可能破坏习惯或链接时，用简短指引替代旧内容。
5. 确认独有知识已经保存后，再删除过期副本。
6. 在人类入口和协作入口中加入文档导航。

条件允许时保留 Git 历史，但不要为了减少文件变动而继续保留误导性内容。

## 反模式

- README 从某个实现细节开始，而不是项目定位和快速开始。
- 把聊天记录、devlog 或 changelog 当成当前规范。
- 用一份巨型文档混合安装、业务规则、schema 和生产恢复。
- 因为模板列出了文件，就创建大量空文档。
- 把同一条规则复制进 README、AGENTS、架构文档和 changelog。
- 文档宣称存在代码实际上没有强制执行的安全行为。
- 把只恢复部分资源描述成完整回滚。
- 只列环境变量名，不写默认值、适用环境和敏感性。
- 在示例中发布基础设施密钥或凭证。
- 给单人项目加入不会维护的流程仪式。
- 修改代码后不更新负责该行为的长期文档。
- 发生实质写入却没有新增或更新本次 changelog。
- 把传统 Git Flow 当成所有项目的默认答案。
- 让多个写入型 agent 任务直接修改同一工作目录。
