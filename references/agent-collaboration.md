# AI 编程助手协作入门

这份参考帮助缺少开发经验的人安全地使用 Codex、Claude Code、GitHub Copilot 等 coding agent。重点是建立简单、可重复的协作习惯，不要求一开始理解所有工具和术语。

## 学习路线

1. 先学会描述一个小任务并检查结果。
2. 再理解上下文和项目规则，减少 agent 猜错。
3. 重复流程出现后再学习 Skill、Plugin 和外部连接。
4. 涉及写入、联网和生产环境前理解权限、沙盒和审批。
5. 手动流程稳定后再使用 automation 和 Hook。
6. 需要并行或团队协作时再学习 thread、branch、worktree、PR 和 CI。
7. 出现问题时按层排查，不要只反复改 prompt。

## 最简单的工作方式

每个任务都按下面顺序进行：

1. **先读**：让 agent 阅读项目说明、相关代码和 Git 状态，先解释它看到了什么。
2. **说清目标**：一次只解决一个具体问题，写明什么算完成、哪些内容不要改。
3. **确认风险**：删除文件、数据库迁移、部署、生产环境和密钥操作必须先说明影响。
4. **小步修改**：只修改与目标有关的文件，不顺手重构无关代码。
5. **实际验证**：运行与改动有关的测试、构建或页面检查，不能只说“应该没问题”。
6. **检查交付**：让 agent 说明改了什么、验证了什么、什么没有验证、下一步是什么。

新手不确定怎样提问时，可以使用：

```text
先只读检查这个项目，不要修改文件。请用容易理解的中文告诉我：
1. 项目做什么；
2. 怎样启动；
3. 当前问题可能在哪里；
4. 你建议的修改步骤和风险。
```

确认方案后再明确要求修改。

## 怎样与 agent 讨论方案

好的协作不是让 agent 替你决定一切，也不是每一步都停下来问。普通、可逆的小细节可以让 agent 遵循项目现有做法直接完成；会影响长期方向或造成明显风险的选择，应先讨论。

以下情况要求 agent 先给方案，不要立即修改：

- 更换技术栈、数据库、部署方式或核心依赖。
- 修改数据结构、权限、安全、登录、支付或对外 API。
- 引入新的 Git 流程、CI/CD、Plugin、Hook、automation 或外部服务。
- 删除大量文件、覆盖已有修改、迁移真实数据或操作生产环境。
- 多种方案都能实现，但成本、风险和后续维护差异明显。

可以要求 agent 使用这个格式：

```text
先不要修改。请根据仓库证据说明：
1. 你发现了什么问题；
2. 为什么需要现在决定；
3. 有哪些可行方案，各自的优点、缺点和风险；
4. 你推荐哪个，为什么；
5. 选定后会修改什么、怎样验证和恢复。
```

如果看不懂方案，直接要求解释术语、给具体例子或说明“什么都不做会怎样”。agent 应给出推荐，但最终选择仍由开发者确认。确认后让 agent 重述范围，再开始高影响操作。

## 理解上下文、token 和成本

- token 是模型处理文字和代码的计量单位，不等同于字符数或字数。
- 上下文是 agent 当前能看到的信息，可能包括聊天记录、项目规则、读取过的文件、工具说明和命令输出。
- 上下文越多不代表效果越好。无关代码、超长日志和重复说明会挤占真正重要的信息。
- 稳定规则写进项目文档，当前任务只提供与目标有关的文件和错误信息。
- 长对话发生压缩、换线程或任务移交后，重新确认目标、Git 状态、已完成内容、验证结果和下一步。
- ChatGPT/Codex 产品额度与 OpenAI API token 账单不是同一种计费方式。价格、额度和缓存规则会变化，需要付费决策时查官方页面。

节省上下文的重点不是少说必要信息，而是删除重复、过期和与当前任务无关的内容。

## 给项目留下稳定说明

不要让 agent 每次都依赖聊天记录重新猜项目。

- `README.md` 面向人，说明项目是什么、怎样安装和启动。
- `AGENTS.md` 或项目已有的规则文件面向 coding agent，说明目录、命令、禁止事项、验证和文档同步要求。
- 架构、业务、数据、配置和运维细节放进对应主题文档，不要全部塞进 `AGENTS.md`。
- 个人电脑路径和临时偏好留在本地，不要写进团队共享规则。
- 密钥、token、密码和生产凭据不能写进文档、prompt 或 Git。

项目规则要写具体动作。例如“修改后运行 `npm test`”比“认真检查代码”更有效。

## 区分规则、Skill、Plugin 和外部连接

| 能力 | 解决的问题 | 新手什么时候需要 |
|---|---|---|
| 项目规则文件 | 每次进入项目都必须知道什么 | 出现稳定命令、目录边界和禁区时 |
| Skill | 某一类重复任务应该怎样做 | 同一流程已经重复解释多次时 |
| Plugin | 安装和分发一组 Skill、工具或连接配置 | 需要分享给团队或跨设备安装时 |
| CLI 或脚本 | 确定、重复、可以测试的操作 | 手工命令容易输错或频繁重复时 |
| connector、API 或 MCP | 读取或操作仓库之外的系统 | 确实需要工单、设计稿、知识库等外部数据时 |

优先顺序是：先写清项目规则，再把重复流程做成 Skill；只有需要分发或组合能力时再用 Plugin，需要外部系统时再加 connector、API 或 MCP。能力越多，权限、供应链和排障成本越高。

## 先认识几个必要概念

| 概念 | 简单理解 | 什么时候需要 |
|---|---|---|
| branch | 一条独立的修改线，避免直接弄乱稳定代码 | 持续维护的项目开始做新任务时 |
| PR | 集中查看改动、测试结果并决定是否合并 | `main` 受保护或会自动部署时 |
| worktree | 同一仓库的另一个独立工作目录 | 两个写入任务需要同时进行时 |
| CI | 代码提交后自动运行的检查 | 希望合并前自动阻止明显错误时 |
| Skill | 某类重复任务的可复用操作说明 | 同一种流程已经反复解释时 |
| Hook | 在特定时机自动执行的小检查 | 手动流程稳定后仍经常漏掉同一检查时 |

只学习当前问题需要的概念。一个人串行开发时不需要 worktree；没有稳定手动流程时也不应先做 Hook 或 automation。

## 区分会话和并行工作

- conversation 或 thread 保存一条任务的聊天和上下文，适合同一目标继续推进。
- 新问题、互不相关的探索或需要干净上下文时，创建新的 thread。
- branch 隔离版本历史，worktree 隔离文件目录；它们解决的问题与 thread 不同。
- subagent 是主任务临时分出的协作者，只适合可以独立分析、审查或验证的子问题。
- 多个只读任务可以查看同一仓库；多个写入任务必须使用独立 branch 和 worktree，避免互相覆盖。
- 写入任务形成首个可运行且已验证的提交后，如果仓库已配置远端且该项目已授权远端写入，应主动推送任务 branch 并创建或更新 PR；不要推送空 branch 或未验证的临时提交，只读任务也不需要为了流程形式创建远端 branch。
- 跨模块、长时间或要求持续做到跑通的任务，主任务在实现后必须自主安排一次隔离的只读验收。优先使用 subagent；只有 chats/threads 可用或用户明确要求时，由主任务自行创建新 thread，不让用户代为协调。
- 独立验收者检查目标、验收标准和真实产物，不继承实现者的结论；主任务仍需亲自复现核心流程并完成最终端到端验收。

不要为了“并行看起来更快”拆分强依赖任务。协调和合并的成本可能高于串行完成。

## 安全边界

让 agent 可以在明确范围内自主完成普通检查和修改，但以下操作必须由人确认目标和影响：

- 删除或批量移动文件。
- 安装、删除或大规模升级依赖。
- 修改数据库结构或真实数据。
- 修改登录、权限、支付和账单逻辑。
- 推送代码、合并 PR、发布或部署。
- 访问生产服务器、外部系统或用户数据。
- 读取、复制或发送密钥和凭据。

权限越高，任务范围应该越小。沙盒和审批只能限制 agent 能碰什么，不能保证它写出的业务逻辑一定正确；最终仍需要测试、代码检查和必要的人工验证。

### 怎样理解沙盒和审批

- 沙盒规定 agent 可以读写哪些文件、是否能联网和能否访问工作区之外。
- 审批规定 agent 越过边界或执行高风险操作时，是否需要人确认。
- 新项目先从只读检查开始；需要修改时开放工作区写入，不要因为嫌麻烦直接给全盘权限。
- 联网时确认目标域名和用途，不把 `.env`、cookie、密钥、私有代码或用户数据发送给不可信服务。
- 具体权限名称和默认值因产品与版本而异，配置前查官方文档和当前客户端。

## Automation 和 Hook

Automation 是按时间或事件重复运行一个任务，适合定期死链检查、CI 失败摘要、提醒和只读报告。第一次配置时写清目标、频率、输入、输出、超时、失败通知和人工复核点，并先观察一次真实结果。

Hook 是在会话开始、工具执行前后或交付前插入的小检查，适合加载必要规则、拦截危险命令、检查敏感信息或提醒补测试。

不要让 Automation 或 Hook 自动执行发布、部署、数据库迁移和不可逆操作。Hook 应短小、可审查，不能替代测试、CI、代码 review 和人工审批。

## 配置文件怎样管理

- 全局配置保存个人跨项目偏好；项目配置只保存团队共同需要的设置；临时要求留在当前任务。
- 配置文件可以保存模型、权限、工具和 profile 等设置，但密钥应放在受控环境变量或 secret manager 中。
- `.env` 通常保存应用运行需要的环境变量，不能与 agent 自身配置混为一谈。
- 新手先使用工具默认值，只在明确遇到问题时修改一个配置，并验证修改是否生效。
- 团队共享命令、目录和安全规则，个人路径、token、私有服务地址和本机偏好不要提交。
- Codex 的 `config.toml`、其他工具的配置文件及字段会变化，具体写法以官方文档为准。

## 什么时候再增加工具

| 已经出现的问题 | 再考虑增加 |
|---|---|
| 每次都要重复说明项目目录、命令和禁区 | 更新项目规则文件 |
| 同一种文档、审查或发布流程反复执行 | Skill 或本地脚本 |
| 多个写入任务互相覆盖文件 | 独立 branch 和 worktree |
| `main` 合并后会自动部署 | PR、CI 和 branch protection |
| 总是漏掉同一个简单检查 | CI、pre-commit 或轻量 Hook |
| 必须读取外部系统中的结构化数据 | 官方 API、CLI 或可信 connector |

先确认问题真实存在，再引入工具。不要因为某个产品提供了 Plugin、MCP、subagent、automation 等功能，就全部装进一个小项目。

## 团队协作

团队开始使用 coding agent 时，先统一最少的共同规则：项目入口、常用命令、目录边界、测试要求、安全红线和 PR 交付内容。PR 中说明 agent 参与范围、实际验证、风险和未确认项，但不要把“由 AI 生成”当成质量保证或质量缺陷的唯一判断。

把反复出现的问题沉淀到合适位置：重复背景进入项目规则，重复流程进入 Skill 或脚本，重复漏检进入 CI，常见故障进入排障手册。成功和失败都应复盘，但聊天记录本身不是长期规范。

## 分层排障

Agent 表现异常时按顺序检查：

1. **目标**：任务、范围、完成标准和禁止事项是否清楚。
2. **事实**：代码、README、项目规则和配置是否互相一致。
3. **上下文**：关键文件是否读取，是否混入旧任务或超长日志。
4. **能力**：是否选错 Skill、Plugin、CLI、connector 或浏览器。
5. **权限**：沙盒、网络、登录和组织策略是否允许当前操作。
6. **工作区**：branch、worktree、未提交文件和并行任务是否冲突。
7. **环境**：依赖、runtime、端口、进程和外部服务是否正常。

要求 agent 引用它依据的文件和命令输出，并区分“已确认”“推测”“无法访问”。测试跑不起来时先区分环境问题和代码问题；遇到阻塞应记录证据，不要通过无关改动掩盖。

## 工具只是例子

不同产品可能使用不同规则文件和配置入口。例如 Codex 常用 `AGENTS.md`，Claude Code 使用 `CLAUDE.md`。项目文档应先描述“项目规则”“验证命令”“并行工作目录”等通用职责，再写当前项目采用的具体工具。

产品功能和默认值会变化。需要记录某个产品的当前行为时，查看它的官方文档和本地实际配置，不能只依赖第三方教程。
