# 可调整的文档大纲

把以下内容当作大纲，不要当作强制模板。删除不相关章节，保留项目中更成熟的现有约定。

## README

```markdown
# 项目名称

用一段话说明项目做什么、谁使用，以及主要运行形态。

## 快速开始

列出前置条件和版本。

1. 安装依赖。
2. 根据示例创建本地配置。
3. 按顺序启动需要的进程。
4. 打开预期 URL 或执行成功检查。

## 文档导航

| 文档 | 用途 |
|---|---|
| ... | ... |

## 常用命令

| 命令 | 用途 |
|---|---|
| ... | ... |

## 重要约束

只保留高影响提醒，并链接到负责详细规则的长期文档。
```

把项目定位和快速开始放在领域细节前面。安装命令必须与仓库中的 lockfile 和 CI runtime 一致。

## 协作说明

使用 `AGENTS.md` 保存 coding agent 指令，使用 `CONTRIBUTING.md` 保存面向参与者的协作流程。不要在两者之间复制相同规则。

```markdown
# 协作原则

- 用户的路径、命令、顺序或技术判断明显有误时，先说明证据、影响和正确做法，不要机械执行。
- 架构、数据、安全、部署、成本或长期流程存在多种合理方案时，提供方案、利弊和推荐理由，与开发者讨论确认后再行动。
- 删除、迁移、发布、生产环境和外部系统写入等高风险操作，先确认目标、备份和恢复方法。
- 已有约定明确、范围内且可逆的普通实现细节，可以说明假设后直接推进，不要让开发者反复确认无关小事。

## 仓库地图

| 路径 | 职责 |
|---|---|
| ... | ... |

## 阅读路径

| 修改类型 | 先读什么 |
|---|---|
| ... | ... |

## 文档同步约定

- 架构变化 -> 架构文档。
- 业务行为变化 -> 业务规则和用户可见说明。
- schema 变化 -> 数据模型和迁移说明。
- 环境变量变化 -> 示例和配置文档。
- 运维变化 -> runbook。
```

规则必须可以直接执行。除非发布自动化会更新，否则不要手工维护“当前版本”。

## 主题文档

```markdown
# 主题

用一句话说明本文负责什么、不负责什么。

## 当前行为

描述已经验证的行为和重要不变量。

## 修改约束

说明协作者必须保留的规则。

## 验证

列出命令或可观察检查。

## 相关文档

使用链接，不要复制内容。
```

精确映射、权限、字段、环境变量和命令对比使用表格；理由和约束使用文字说明。

## 配置说明

```markdown
# 配置

说明怎样创建本地配置以及怎样处理密钥。

| 变量 | 默认值 | 必填环境 | 敏感 | 用途 |
|---|---|---|---|---|
| ... | ... | ... | 是/否 | ... |

## 生产环境要求

区分警告、默认值和强制启动失败。

## 修改检查

1. 配置加载器。
2. 环境变量示例。
3. 本文档。
4. CI/CD、进程管理器和部署环境。
```

示例中不能出现真实密钥。明确标记会进入客户端构建产物的变量。

## 运维 Runbook

```markdown
# 运维

## 前置条件

说明权限、平台、runtime、备份和目标确认。

## 部署

按顺序列出命令或自动化步骤，并写明成功信号。

## 验证

列出健康检查、smoke test、日志和用户可见检查。

## 回滚

准确说明恢复哪些内容、不恢复哪些内容、触发条件、负责人和验证方式。

## 故障排查

现象 -> 诊断命令 -> 结果解释 -> 恢复操作。
```

只有应用代码、依赖、配置、数据兼容性和必要资源都被覆盖时，才能称为完整回滚。

## Changelog 记录

```markdown
# 修改标题

## 日期

YYYY-MM-DD

## 问题

说明为什么需要修改。

## 改动

说明修改了什么。

## 验证

说明实际检查了什么。

## 影响范围

列出受影响的用户、模块、数据、配置或部署。
```

仓库采用一项修改一个文件时，优先使用 `YYYY-MM-DD-topic.md` 这种日期在前的文件名。changelog 只保留历史，同一任务还要修正长期主题文档。
