# 帮助新手建立项目文档与协作体系

## 默认介入原则

把新手或主要依赖 AI 编程助手的用户提出的项目相关问题，默认视为本 Skill 的适用场景。不要等到用户明确提到“文档”“协作”或“方案”才使用它。

- 用户提出功能、页面、产品改进、重构、修复、部署、运行、报错、配置、模型、插件、Git、自动化、Agent、Skill、上下文或权限问题时，先使用本 Skill 的调查和解释方式。
- 用户说“我想做”“能不能”“为什么”“怎么改”“给个方案”“像某个产品一样”时，先确认真实项目状态，用新手能理解的语言说明当前可做的第一步，再按请求类型继续实施或解释。
- 即使任务最终只改一行代码，也要先读取适用的仓库说明、保留用户改动，并在完成后说明怎么验证；只有与项目无关的闲聊、通用知识问答，或用户明确要求跳过引导时才不使用。
- 不要借此扩大任务范围：使用本 Skill 是为了让工作可理解、可验证和可维护，不是每次都创建文档、要求用户选择方案，或加入复杂流程。

目标不是堆积文档或套用专业团队的复杂流程，而是让缺少经验的开发者能够回答：项目做什么、怎样运行、改哪里、怎样验证、哪些操作有风险。建立最小但完整的体系，让新参与者不依赖聊天记录或某个人的记忆，也能继续维护项目。

## 选择工作模式

- 用户要求审计、评审、解释或报告状态时，只检查并报告，不修改文件。
- 用户是在学习概念或询问“这是什么、什么时候用”时，先按新手能理解的顺序解释并给一个小例子，不要擅自改仓库或一次抛出整套进阶配置。
- 用户要求建立、修复、重构或维护文档时，直接更新仓库并验证结果。
- 执行代码修改时，把相关文档更新视为同一项工作，不要把必需的配置、示例和文档拆到以后再做。
- 用户给出的路径、命令、操作顺序或技术判断明显有误时，先用易懂语言指出问题，说明可行方案及利弊；选择会造成破坏、扩大范围或长期影响时，与开发者讨论并确认后再行动。

## 与开发者讨论后再决定

不要把新手排除在重要决策之外。出现以下情况时，暂停写入操作，先讨论：

- 存在两种以上合理方案，而且会明显影响架构、数据结构、安全、部署、成本、外部服务或长期维护。
- 需要新增长期流程或基础设施，例如复杂 Git workflow、CI/CD、数据库迁移、权限体系、Plugin、Hook 或 automation。
- 操作难以恢复、会影响生产环境、会写入外部系统，或者可能删除、覆盖用户数据和现有修改。
- 用户目标、仓库现状和已有规则互相冲突，无法通过只读调查确定正确方向。
- 用户明确要求先讨论、比较方案或不要立即修改。

讨论时按以下顺序表达：

1. 说明发现的问题和仓库证据，不只给结论。
2. 用新手能理解的语言说明为什么需要决定，以及不处理的影响。
3. 提供二到三个真实可行的方案，分别说明收益、代价、风险和适用条件。
4. 明确给出推荐方案和推荐理由，但不要把推荐伪装成唯一答案。
5. 说明确认后会修改什么、不会修改什么，以及怎样验证和恢复。
6. 提出一个清楚的确认问题，等待开发者选择后再执行高影响操作。

不要把所有实现细节都变成确认题。只读调查、已有约定明确的做法、范围内的小型可逆修改和不会改变对外行为的普通实现细节，可以说明假设后直接推进。无法判断是否属于重大选择时，优先讨论。

## 执行流程

### 1. 写文档前先调查

1. 阅读 `AGENTS.md`、`CONTRIBUTING.md` 和作用于当前目录的其他仓库指令。
2. 检查 Git 状态，保留用户已有修改。
3. 先确认项目类型、主要入口、依赖清单、启动命令、配置方式和部署方式，再按实际需要盘点 README、`docs/`、changelog、ADR、schema、路由、测试和运维脚本。
4. 阅读真正控制行为的代码和配置。把现有文字视为待验证的说法，不能直接当成事实。
5. 识别真实读者：首次参与的开发者、维护者、运维人员、非技术用户和自动化 agent。只为实际存在的读者建文档。

建立或重构体系时，阅读 [references/documentation-system.md](references/documentation-system.md)。

### 2. 诊断文档与协作缺口

重点检查：

- 没有明确的人类入口或快速开始。
- README、聊天记录、devlog 和 changelog 同时充当事实来源。
- 业务规则埋在组件、prompt 或注释中。
- 代码读取的环境变量没有进入示例和配置说明。
- 部署、备份和回滚说明与自动化脚本不一致。
- 数据字段缺少类型、归属或迁移说明。
- 命令已经过期、具有破坏性、依赖特定平台或从未验证。
- 文档围绕文件堆放，而不是围绕协作者需要回答的问题组织。
- 使用陌生术语，却没有解释它解决什么问题以及读者下一步要做什么。
- 默认新参与者已经理解 Git、branch、PR、部署和回滚。
- 为简单项目加入大量模板、审批、branch 层级或自动化，维护成本高于实际收益。
- 项目使用 coding agent，却没有告诉它先读什么、哪些文件不能随意修改、完成后怎样验证。

用清楚的中文解释重要问题。首次出现陌生术语时，用一句话说明它是什么、为什么当前项目需要它。遇到破坏性或生产环境操作时，先纠正风险假设，与开发者确认目标、备份和恢复方式后再执行命令。

### 3. 设计最小可行体系

现有命名和约定合理时继续沿用。只有仓库中确实存在需要长期归属的事实或流程时才新建文档。按下面三个层级逐步建设，不要一次性全部创建。

1. **基础层**：先让项目能够被理解和运行。通常包括 README、配置示例、必要的协作入口和文档导航。
2. **项目层**：项目确实存在业务规则、数据、部署或用户操作时，再建立对应主题文档。
3. **进阶层**：出现多人协作、自动部署、并行任务或重复自动化需求后，再引入 PR、CI、worktree、ADR、Skill、Hook 等机制。

默认归属如下：

- `README.md`：项目定位、前置条件、快速开始、常用命令和文档导航。
- `AGENTS.md` 或 `CONTRIBUTING.md`：协作规则、仓库地图、阅读路径和文档同步约定。
- 架构文档：模块边界、状态与请求流程、存储选择和非显然约束。
- 业务规则：权限、不变量、计算、导出和业务流程行为。
- 数据模型：持久化实体、字段、关系、标识符和迁移。
- 配置文档：环境变量、默认值、生产必填项、密钥处理和外部服务。
- 运维文档：CI/CD、部署、备份、回滚、健康检查、日志和恢复边界。
- 用户指南：真实用户流程和权限，仅在非技术用户实际操作产品时创建。
- changelog：记录某次修改，不作为当前行为的唯一来源。
- ADR：记录需要长期保留取舍理由的重要架构决策。
- Git 协作说明：branch、worktree、PR、合并、发布和 hotfix 规则，可放在 `CONTRIBUTING.md`、`AGENTS.md` 或运维文档中。

不要创建空占位文档，也不要把同一条规则复制到多个文件；在其他位置链接到负责该事实的文档。

项目缺少 Git 协作规则、存在自动部署或需要并行任务时，阅读 [references/git-workflow.md](references/git-workflow.md)。根据团队规模和发布方式选择最小流程，不要默认套用传统 Git Flow。

项目由新手配合 coding agent 开发，用户询问上下文、项目规则、Skills、Plugins、权限、automation、Hooks、沙盒、thread、配置、团队实践或排障，或者用户不知道怎样安全地让 AI 修改项目时，阅读 [references/agent-collaboration.md](references/agent-collaboration.md)。先建立最简单的“先读、再改、验证、说明”习惯；只有重复需求已经出现时，才介绍进阶机制。

### 4. 面向缺乏协作经验的开发者实施

明确给出第一条操作路径：

1. 说明项目做什么。
2. 说明前置条件，不假设读者已经知道本地工具链。
3. 按正确顺序提供可直接执行的安装和启动命令。
4. 说明预期 URL、进程和成功信号。
5. 给出最短的成功路径，再提供按问题查阅的详细文档；不要要求新手一开始读完整套体系。
6. 把每类修改路由到修改前必须阅读、修改后必须同步的文档。
7. 区分开发默认值与生产要求。
8. 在命令前说明风险、备份方式和失败后的恢复方法。
9. 只有当前任务确实用到时，才解释 branch、PR、worktree、CI 等概念，并给出当前项目可执行的下一步。

创建或大幅改写文档时，参考 [references/templates.md](references/templates.md)。保留有价值的旧内容，把旧笔记迁移到长期文档；旧路径很可能仍被引用时，保留简短兼容指引。

不要在文档中写入凭证、token、private key、密码或生产密钥。只记录变量名以及申请、配置或轮换方法。

### 5. 同步维护

在同一个任务中执行对应更新：

- 架构或跨模块流程变化：更新架构文档。
- 业务行为、权限、校验或导出变化：更新业务规则，并同步用户可见说明。
- 字段、schema、集合或迁移变化：更新数据模型。
- 环境变量变化：同时更新环境变量示例和配置文档。
- CI/CD、部署、备份、回滚或监控变化：更新运维文档。
- 用户操作流程变化：更新用户指南。
- branch、PR、worktree、发布或保护规则变化：更新 Git 协作说明和相关自动化。
- 项目已经使用 coding agent 规则、Hook 或 automation 时：对应更新协作说明和实际配置；项目未使用时不要为了完整性强行创建。
- 仓库约定需要 changelog 时记录本次修改，同时更新负责当前行为的长期文档。

除非发布工具会自动更新，否则不要在文档中硬编码“当前版本”。

### 6. 验证事实与导航

存在 Python 时运行审计脚本：

```bash
python scripts/audit_docs.py <repository-root>
```

从本 Skill 目录定位脚本，不要假设脚本位于目标仓库。脚本检查本地 Markdown 链接，并比较常见环境变量读取点与仓库中的环境变量示例。

还要人工验证：

1. 按风险比例运行或检查文档中的安装、构建、测试、迁移和健康检查命令。
2. 比较 API 摘要与实际路由注册。
3. 比较配置文档与代码、工作流和环境变量示例。
4. 比较部署、备份和回滚说明与自动化脚本的准确步骤。
5. 从链接所在文件出发验证相对路径。
6. 项目已经采用 Git 协作流程时，比较说明与 branch protection、CI 和部署触发条件。
7. 存在 Git 时运行仓库格式检查和 `git diff --check`。

只检查了仓库文件时，不要声称已经验证生产环境。明确说明证据边界。

### 7. 清楚交付

先说明形成了什么文档与协作体系，再总结：

- 入口和单一事实来源的归属。
- 修正的重要不一致。
- 新手接下来最应该做的一到三个具体动作。
- 项目已经采用 Git 或 coding agent 协作规则时，说明选择理由；未采用时不要制造必须立即配置的待办。
- 已完成的验证和没有运行的检查。
- 仍需开发者、运维人员或生产权限确认的事实。

不要用流程仪式压垮缺乏经验的开发者。只有项目确实需要时，才加入模板、ADR 系统、PR 清单、release branch 等机制。
